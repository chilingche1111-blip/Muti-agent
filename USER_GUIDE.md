# Context Atlas 使用说明书

> 适用对象：业务用户、测试人员、开发人员和本地部署管理员
> 默认地址：`http://127.0.0.1:8080/`
> 验证中心：`http://127.0.0.1:8080/test.html`

---

## 1. 产品用途

Context Atlas 是一个面向企业长文档的多 Agent 研究系统，主要提供以下能力：

- 像普通智能体一样进行通用问答、写作、解释和代码辅助；
- 上传 PDF、Word、PowerPoint 和文本资料；
- 对超过本地模型 64K 上下文窗口的资料进行事实抽取、总结、风险审查和比较；
- 精确读取全文或指定页码；
- 可选联网检索并显示网页来源；
- 查看证据引用、Agent 执行轨迹和上下文占用；
- 在受保护的验证中心运行 64K+ 测试。

系统不会修改底层模型的原生窗口，而是通过多个独立 Agent 分担任务，让每次模型调用都低于 64K。

## 2. 快速开始

### 2.1 安装依赖

进入项目目录：

```bash
cd long-context-research-agent
python3 -m pip install -e .
```

要求：

- Python 3.9 或更高版本；
- 系统已安装 `curl`；
- 如需解析旧版 `.doc`、`.ppt`，建议安装 LibreOffice。

### 2.2 启动服务

```bash
python3 -m long_context_agent.web_server --host 127.0.0.1 --port 8080
```

看到以下信息代表启动成功：

```text
Context Atlas: http://127.0.0.1:8080
测试中心: http://127.0.0.1:8080/test.html
```

浏览器打开：

```text
http://127.0.0.1:8080/
```

### 2.3 停止服务

如果服务运行在当前终端，按：

```text
Ctrl + C
```

## 3. 主界面说明

主界面分为三个区域。

### 3.1 左侧：资料与模型配置

左侧包含：

1. 文档上传区；
2. 文档 Token、Chunk、Section 和大小统计；
3. 模型连接配置；
4. 当前文档是否超过 64K 的提示。

### 3.2 中间：回答和执行结果

提交问题后，中间区域显示：

- 用户问题；
- LLM 或多 Agent 回答；
- Validator 状态；
- 长上下文隔离指标；
- 原文或网页引用；
- Agent 图执行轨迹。

回答区域支持独立滚动，长文本不会把页面锁死。

### 3.3 底部：回答控制和输入框

底部包含：

- 回答范围：自动、通用、文档；
- 联网检索开关；
- 快捷问题；
- 问题输入框；
- Worker 数量和 Reducer 扇入等运行参数。

## 4. 配置企业 LLM API

### 4.1 打开模型连接

在左侧点击“模型连接”，填写：

| 字段 | 说明 | 示例 |
|---|---|---|
| Base URL | OpenAI 兼容接口根地址，通常以 `/v1` 结尾 | `https://llm.company.example/v1` |
| API Key | 公司分配的访问密钥 | 不要写入代码或截图 |
| 模型 | 网关支持的模型名称 | `qwen` |
| 超时 | 单次请求最大等待秒数 | `90` |
| 档案名称 | 当前配置的本地名称 | `公司 Qwen` |

### 4.2 测试连接

点击“测试连接”。系统会向模型发送一个极小请求，要求只回复 `OK`。

连接成功后：

- 状态变为“连接正常”；
- 当前设置可用于正式智能问答；
- 验证中心可以复用同一个已验证 API 临时句柄。

连接失败时，先查看页面提示，再参考第 13 节故障处理。

### 4.3 保存和切换配置

填写档案名称后点击“保存会话配置”。之后可以从“配置档案”下拉框切换。

注意：

- 配置保存在当前浏览器会话中；
- 服务端验证句柄保存在内存中；
- 服务重启后需要重新测试连接；
- 生产环境应改用企业密钥管理系统。

## 5. 上传和解析文档

### 5.1 支持的格式

| 类型 | 扩展名 | 主要解析内容 |
|---|---|---|
| PDF | `.pdf` | 文本层和页码 |
| Word | `.docx` | 标题、段落和表格 |
| PowerPoint | `.pptx` | 幻灯片文字和演讲者备注 |
| 旧版 Office | `.doc`、`.ppt` | 本地转换后解析 |
| 文本 | `.txt`、`.md`、`.csv`、`.json`、`.html`、`.xml`、`.rtf` | 文本内容 |

单文件最大 20 MB。

### 5.2 上传步骤

1. 点击“选择或拖入文档”。
2. 选择本地文件，或把文件拖到上传区域。
3. 等待状态显示“已索引”。
4. 检查 Token、Chunk、Section 和大小统计。

如果 Token 超过 65,536，页面会提示资料将由多个隔离 Agent 分担。

### 5.3 扫描 PDF

如果 PDF 是扫描图片且没有文本层，系统会提示需要 OCR。这种文件不会直接进入 Agent 问答，以免模型在没有真实文本时生成无来源答案。

建议先使用 OCR 工具生成带文本层的 PDF，再重新上传。

## 6. 选择回答范围

### 6.1 自动

适合大多数场景。

- 没有上传文档时：使用通用 LLM；
- 已上传文档时：文档问题优先进入多 Agent；
- 明显的普通聊天、写作或解释：仍可以直接使用 LLM。

### 6.2 通用

强制使用底层 LLM 的普通能力，不要求文档引用。

适合：

- 普通聊天；
- 写作和改写；
- 翻译；
- 概念解释；
- 编程辅助。

系统保留最近最多 20 条消息，并把通用对话输入控制在约 24K Token 内。

### 6.3 文档

强制基于已上传资料运行长文档 Agent。

适合：

- 提取关键事实；
- 总结全文；
- 查找风险、矛盾和缺口；
- 比较不同方案或章节；
- 查询指定页面；
- 需要原文引用的回答。

如果没有上传文档，文档模式会提示先导入资料。

## 7. 常用提问方法

### 7.1 事实提取

```text
这个项目的预算、负责人和验收日期分别是什么？
```

```text
请提取所有产品名称、规格和对应价格，并附原文依据。
```

### 7.2 全文总结

```text
请总结这份资料的背景、核心内容、主要结论、行动项和限制，并给出引用。
```

全文总结会由多个 Analysis Agent 分别覆盖文档开头、中部和末尾，再通过 Reducer 合并。

### 7.3 风险和矛盾

```text
请找出文档中相互矛盾、条件缺失或需要进一步核实的信息。
```

### 7.4 比较分析

```text
请从成本、周期、风险和适用范围四个维度比较方案 A 与方案 B。
```

### 7.5 读取全文

```text
请输出 PDF 全部内容。
```

系统直接返回解析文本。长文档通过“上一段/下一段”分页，每段约 12,000 字符，不调用 LLM。

### 7.6 指定页码

读取原文：

```text
请输出第 16–20 页的内容。
```

分析页面：

```text
请总结第 16–20 页的核心观点和风险。
```

前一种请求直接读取页面；后一种请求先精确截取目标页，再建立临时索引并调用多 Agent。

## 8. 使用联网检索

### 8.1 开启方式

在输入框上方打开“联网检索”。系统最多检索 5 个公开网页结果。

### 8.2 通用问答加联网

```text
请联网查询 LangGraph 最近的稳定版本和主要变化，并给出来源。
```

搜索结果会先提供给 LLM，回答使用 `[网页1]` 等编号标注来源。

### 8.3 文档问答加联网

```text
请比较当前内部方案与公开资料中的行业做法，并区分内部文档和外部来源。
```

系统会把网页摘要作为外部资料加入临时索引，并在引用区域显示可点击链接。

### 8.4 企业 SearXNG

如公司部署了 SearXNG，可在启动服务前设置：

```bash
export CONTEXT_ATLAS_SEARXNG_URL="https://search.company.internal"
```

未配置时，默认使用 Bing HTML 搜索。

注意：联网检索使用的是搜索结果摘要，不等于完整网页。重要结论应点击来源进一步确认。

## 9. 如何理解回答页面

### 9.1 回答状态

| 状态 | 含义 |
|---|---|
| 直接回答 | 通用 LLM 已完成回答 |
| 原文直接读取 | 结果来自已解析文档，没有调用 LLM |
| Validator 通过 | 文档 Agent 的硬校验和语义校验同时通过 |
| Validator 拒绝 | 至少一项任务、证据、矛盾或质量检查未通过 |

### 9.2 原文引用

引用卡片包含：

- 文档名或网页标题；
- 章节、页码或来源类型；
- Evidence ID；
- 原文摘录；
- 网页 URL（如果是联网来源）。

### 9.3 多 Agent 上下文隔离证明

| 指标 | 含义 |
|---|---|
| 完整资料 | 上传文档的估算总 Token |
| 最大单次 Prompt | 本次所有 Agent 中最大的输入 |
| 64K 窗口占用 | 最大 Prompt 占模型窗口的比例 |
| 子任务 | 主 Agent 创建的任务数量 |
| Agent 隔离 | 专业 Agent 是否使用独立消息上下文 |
| 主 Agent 接收原文 | 应当显示“否” |
| 模型调用 | 本次使用的模型请求数量 |

真正证明“处理超过64K”的条件是：

1. 完整资料确实超过 65,536 Token；
2. 所有单次 Agent 调用均未越界；
3. Validator 批准结果；
4. 用例答案和引用完整。

### 9.4 执行轨迹

展开轨迹可以看到：

- 主 Agent 如何规划任务；
- 任务进入哪个专业 Agent；
- 使用了何种检索策略；
- Reducer 执行了多少层；
- Validator 是否通过；
- 是否发生定向重规划。

## 10. 验证中心

验证中心用于测试和验收，不是正式业务问答入口。

地址：

```text
http://127.0.0.1:8080/test.html
```

本地默认账号：

```text
用户名：tester
密码：123456
```

### 10.1 离线流程演示

使用固定的 `DeterministicTestModel` 验证：

- LangGraph 节点和条件边；
- Supervisor 任务拆分；
- Specialist 上下文隔离；
- Reducer 归并；
- Validator 和引用闭环；
- 64K+ 文档的首、中、尾事实。

离线流程模型不具备任意业务文档理解能力。

### 10.2 真实 API 验收

1. 先在主界面完成 API 连接测试。
2. 进入验证中心并登录。
3. 选择真实 API 模式。
4. 设置 Worker、Reducer 扇入和最大重规划次数。
5. 点击开始验收。

验证中心复用主界面验证成功的临时 API 句柄，不单独保存 API Key。

### 10.3 六项验收门禁

1. 测试资料超过 64K；
2. 每次 Agent 调用低于 64K；
3. 专业 Agent 上下文隔离；
4. 主 Agent 不接收完整原文；
5. Validator 硬校验和语义校验通过；
6. 四个用例正确且带引用。

### 10.4 失败原因

失败用例会显示：

- 缺失的预期事实；
- 是否缺少引用；
- 哪一项程序硬校验失败；
- Validator 是否返回非法 JSON；
- 是否存在缺失任务或矛盾；
- 失败是否可重试。

## 11. 命令行测试

### 11.1 自动测试

```bash
python3 -m unittest discover -s tests -v
```

当前应通过 30 项测试。

### 11.2 64K+ 离线基准

```bash
python3 -m long_context_agent.benchmark \
  --document test_data/enterprise_research_128k.md \
  --cases test_data/test_cases.json \
  --mode offline
```

当前基准资料约 177,420 Token，包含文档首部、中部、末部和跨位置联合问题。

### 11.3 真实 API 命令行基准

先设置环境变量：

```bash
export OPENAI_BASE_URL="https://llm.company.example/v1"
export OPENAI_API_KEY="your-api-key"
export OPENAI_MODEL="qwen"
```

然后运行：

```bash
python3 -m long_context_agent.benchmark --mode live
```

## 12. 安全使用要求

1. 不要把真实 API Key 写入 README、源码或 `.env.example`。
2. 不要在截图、群聊或测试报告中暴露密钥。
3. `.env` 已被 `.gitignore` 排除。
4. 曾经暴露的 API Key 应立即轮换。
5. 本地默认测试密码不能用于生产环境。
6. 生产部署应接入 OIDC/LDAP 和企业密钥管理系统。
7. 公共联网检索结果可能不可信，重要信息必须核验来源。

生产环境可覆盖测试账号：

```bash
export CONTEXT_ATLAS_TEST_USER="your-user"
export CONTEXT_ATLAS_TEST_PASSWORD="a-strong-password"
```

## 13. 常见问题与故障处理

### 13.1 页面无法打开

检查端口：

```bash
lsof -nP -iTCP:8080 -sTCP:LISTEN
```

如果没有进程监听，重新启动服务。

### 13.2 LLM 请求失败

依次检查：

1. Base URL 是否包含正确的 `/v1`；
2. API Key 是否有效；
3. 模型名是否由网关支持；
4. 公司网络或 VPN 是否可访问网关；
5. 超时时间是否过短；
6. 网关是否支持 `/chat/completions`。

点击“测试连接”获取更具体的错误。

### 13.3 通用问答提示缺少 API 配置

通用问答必须使用真实企业 LLM。打开左侧模型连接，完整填写 Base URL、API Key 和模型名。

### 13.4 找不到指定页面

原因通常是：

- PDF 没有文本层；
- 解析文本中不存在目标页标记；
- 文档实际页数不足；
- Word 文档没有稳定的物理页码信息。

PDF/PPT 的页码定位最可靠。

### 13.5 回答缺少文档内容

建议：

1. 明确使用“文档”回答范围；
2. 在问题中写明对象、字段或页码；
3. 查看引用是否召回正确章节；
4. 对全文问题使用“总结全文”而不是过于抽象的短查询；
5. 查看执行轨迹中的检索策略和 Validator 说明。

### 13.6 联网检索失败

可能原因：

- 当前网络无法访问 Bing；
- 搜索页面触发验证；
- 企业代理限制外网；
- SearXNG 地址配置错误。

企业环境建议部署内部 SearXNG 并配置 `CONTEXT_ATLAS_SEARXNG_URL`。

### 13.7 验证中心提示 API 句柄过期

返回主界面重新点击“测试连接”。服务重启后，内存中的临时句柄会失效。

### 13.8 Validator 未通过

展开失败原因，判断属于：

- 预期事实缺失；
- 引用缺失；
- Specialist 没有生成 Finding；
- Artifact 无效；
- Reducer 引入未知证据；
- 模型 JSON 格式错误；
- 语义矛盾；
- 上下文预算越界。

系统只会自动重试可恢复任务，不会绕过硬校验。

## 14. 使用边界

- 多 Agent 提高可处理的总资料量，但通常增加总调用次数。
- 系统不能改变底层模型本身的知识和推理上限。
- 文档结论的可靠性取决于文件解析、检索召回和模型能力。
- 离线基准用于证明架构，不代表真实模型的业务准确率。
- 联网搜索摘要可能不完整。
- 当前 Checkpointer、Artifact Store 和 API Profile 主要为内存实现。
- 扫描 PDF 需要外部 OCR。

更完整的架构原理、上下文预算和测试证据请阅读项目根目录的 [README.md](README.md)。

界面颜色、字体、间距和组件规范保存在 `design-system/context-atlas/MASTER.md`。
